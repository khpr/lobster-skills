## G1: 輪詢狀態太頻繁觸發 API rate limit

**錯誤**：洗衣完成通知用 while loop 每秒輪詢 ThinQ API，短時間內大量請求觸發 rate limit，之後 API 呼叫全部失敗。
**正確**：輪詢間隔至少 30 秒。採用 cron job 而非 while loop，避免 session 阻塞。有 webhook 支援時優先用 push 通知而非輪詢。
**觸發情境**：等待洗衣機/烘衣機完成，需要主動通知的場景。

---

## G2: 裝置離線或斷網時未正確處理錯誤

**錯誤**：裝置離線時 API 回傳錯誤，agent 沒有處理就直接回報「狀態：未知」或崩潰。
**正確**：ThinQ API 呼叫加上 try/catch，離線時明確回報「裝置目前離線，請確認網路或裝置電源」，不顯示技術錯誤碼。
**觸發情境**：家中網路異常、裝置暫時斷電。

---

## G3: Access Token 過期未自動刷新

**錯誤**：Token 過期後 API 呼叫開始回 401，agent 繼續重試但不知道要刷新 token，導致持續失敗。
**正確**：收到 401 時先嘗試 token refresh，refresh 失敗才報告需要重新登入。不要在 token 問題未解決時重試超過 2 次。
**觸發情境**：長時間未使用 lg-thinq 後第一次呼叫，或 refresh token 也過期。
